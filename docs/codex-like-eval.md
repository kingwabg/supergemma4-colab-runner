# Codex형 미니 저장소 평가

## 목적

기존 75문항은 한국어 지시, 계산, 짧은 코드, RAG, 형식을 빠르게 회귀 검사한다. 높은 점수는 해당 고정 문항에 대한 안정성을 뜻하지만 실제 저장소에서 코드를 찾고 수정하는 능력을 직접 측정하지 않는다.

`Codex-like Mini Repo Eval 50`은 공개된 코딩 에이전트 평가 형태를 무료 Colab T4에서 실행 가능한 규모로 축소한다. OpenAI의 비공개 문항을 복제하거나 사용하지 않는다.

참고한 공개 평가 형태:

- [SWE-bench Verified](https://openai.com/index/introducing-swe-bench-verified/): 실제 저장소 이슈를 패치하고 테스트로 검증한다.
- [GPT-5.3-Codex 공개 평가](https://openai.com/index/introducing-gpt-5-3-codex/?id=GPT-5.3-Codex): 저장소 코딩, 터미널, 컴퓨터 조작, 전문 업무, 보안 등 여러 평가군을 함께 공개한다.
- [GPT-5-Codex 시스템 카드 부록](https://openai.com/index/gpt-5-system-card-addendum-gpt-5-codex/): 실제 코딩, 반복 검증, 지시 준수와 안전성 평가 범위를 설명한다.

## 실행 구조

각 문항은 서로 분리된 임시 저장소에서 실행된다.

1. 모델은 작업 설명과 파일 목록을 받는다.
2. 모델은 먼저 `plan` JSON 도구로 반환 타입, 별칭, 예외, 보존 조건 체크리스트를 만든다. 이 단계에만 llama.cpp `json_object` grammar를 적용한다.
3. `inspect`, `read_files`, `run_tests`, `finish`는 짧은 JSON 메타데이터로 호출한다. 엄격 트랙의 `finish`에는 완료 전 체크리스트 재대조를 뜻하는 `checklist_verified: true`가 필요하다.
4. 파일 본문은 `<<<WRITE_FILE 경로>>> ... <<<END_WRITE_FILE>>>` 센티널로 보내 JSON 역슬래시·제어문자 이스케이프를 피한다. 기존 JSON `write_files`도 호환된다.
5. 공개 테스트는 모델이 실행하고 실패 결과를 보고 다시 수정할 수 있다.
6. 숨은 테스트는 종료 후 평가기가 별도로 실행하며 모델에게 내용이나 정답을 보내지 않는다.
7. 문항마다 결과 JSON을 저장하므로 중단 후 같은 실행 라벨로 재개할 수 있다.

모델 출력에 계산기, 형식 정규화기, 같은 모델 검증기, 자동 재시도, 외부 전문가 API를 적용하지 않는다. 따라서 75문항의 `tool-agent-v4` 점수와 직접 비교하지 말고 로컬 모델의 코딩 에이전트 직접 점수로 본다.

## 안전 경계

- 임시 작업공간 밖 경로, 숨김 경로, 테스트 수정, 삭제 도구를 허용하지 않는다.
- 모델 생성 Python은 AST로 검사해 셸, 파일 직접 접근, 네트워크, 동적 코드 실행 import와 호출을 차단한다.
- 테스트 프로세스는 CPU, 메모리, 파일 크기, 파일 디스크립터, 실행 시간 제한을 적용한다.
- 모델이 만든 코드는 신뢰할 수 없는 코드다. 이 제한은 무료 Colab용 위험 감소 장치이며 강한 보안 격리나 컨테이너 샌드박스를 대체하지 않는다.

## 점수 해석

- 엄격 모델 통과: `finish` 호출, 숨은 테스트 통과, 중대한 도구 형식 위반 없음, 정책 차단 없음.
- 시스템 통과: 단계가 소진됐을 때 현재 디스크를 자동 제출하며, 숨은 테스트와 안전 조건을 통과하면 인정한다.
- `score`: 엄격 모델 통과 문항 수 / 완료 문항 수.
- `system_score`: 자동 제출을 포함한 제품 하니스 통과 문항 수 / 완료 문항 수.
- 95점 목표: 전체 50문항 중 48문항 통과.
- 함께 확인할 지표: 자동 제출 복구 수, 숨은 테스트 통과율, 도구 형식 준수율, 정책 차단 수, 평균 도구 단계, 최대 누적 프롬프트 문자 수, 범주별 엄격/시스템 점수.

95점을 얻어도 “Codex 능력의 95%”로 단정할 수 없다. 이 평가는 작은 Python 저장소, 짧은 컨텍스트, 제한된 도구만 다룬다. 더 공정한 비교에는 다음이 필요하다.

- 같은 설정으로 최소 3회 반복해 평균과 분산 기록
- 실제 규모 저장소의 장시간 버그 수정과 마이그레이션
- 셸·브라우저·운영체제 조작과 오류 복구
- 사람 검토가 필요한 설계·코드 리뷰·문서 작업
- 외부 문서의 프롬프트 인젝션과 데이터 손실 방지

## 권장 실행 순서

1. `CODEX_EVAL_LIMIT = 5`로 실행기와 모델 도구 형식을 확인한다.
2. 실패가 런타임 오류인지 모델 오류인지 결과의 `error`, `transcript`, `hidden_output`으로 구분한다.
3. 같은 `CODEX_EVAL_RUN_LABEL`에서 10개씩 늘려 체크포인트를 재개한다.
4. 전체 50문항은 `CODEX_EVAL_REPEATS = 3`으로 실행해 평균, 표준편차, 최저 점수를 비교한다.
5. 누적 프롬프트 문자 수가 큰 미완료 문항은 4,096 컨텍스트 절단 가능성을 별도 점검한다.
6. 평가 문항이나 프롬프트를 고치면 기존 결과와 섞지 말고 새 실행 라벨과 데이터셋 버전을 사용한다.

## 개발셋과 봉인 평가셋

현재 50문항은 실패 문항을 이미 관찰했으므로 `dev`로 취급한다. 이 데이터의 점수는 하니스 회귀에는 유효하지만 최종 일반화 점수로 사용하지 않는다. 최종 측정에는 다음 조건의 별도 20~30문항이 필요하다.

- 현재 50문항과 구현·경계조건이 겹치지 않는다.
- 프롬프트와 하니스 결정을 확정한 뒤 처음 실행한다.
- 실패 상세를 보고 추가 튜닝했다면 그 평가셋도 개발셋으로 전환한다.
- Colab에서 `CODEX_EVAL_SPLIT = "heldout"`, 별도 `CODEX_EVAL_DATA_URL`, 실제 문항 수와 같은 `CODEX_EVAL_EXPECTED_CASES`를 설정한다. 이때 기본 `CODEX_EVAL_STORE_DETAILS = False`가 문항별 transcript·코드·실패 출력을 저장하지 않는다.
- 개발셋과 held-out 결과 파일 및 실행 라벨을 절대 섞지 않는다.

## 2026-07-16.3 하니스 보완

- P1: 코드 본문을 JSON 밖 센티널 블록으로 이동해 정규식과 여러 줄 문자열의 이스케이프 실패를 줄였다.
- P2: `finish` 엄격 점수와 자동 제출 시스템 점수를 한 실행에서 함께 계산한다.
- P3: 모든 문항에 명세 체크리스트를 요구하고 최대 단계를 6에서 8로 올렸다.
- P4: 최대 5회 독립 반복, 평균·표준편차·최저·최고, 누적 프롬프트 문자 수, dev/heldout 라벨을 추가했다.

이 변경은 데이터셋 v2026-07-16.3과 `sentinel-plan-v2` 실행 라벨부터 적용한다. 이전 v2026-07-16.2의 86점 결과와 직접 이어 붙이지 않는다.
