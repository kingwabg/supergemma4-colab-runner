# Implementation Plan: Spec-driven Quality Agent

## 아키텍처

- 기존 GGUF 로더와 채팅 함수는 유지한다.
- `supergemma_agent/spec_workflow.py`가 사양 산출물 생성과 단계별 결정적 검사를 담당한다.
- `supergemma_agent/evaluation.py`가 평가셋 검증, 채점, 체크포인트, 범주 요약을 담당한다.
- 노트북은 GitHub 저장소를 얕게 동기화해 두 모듈과 평가 JSON을 가져온다.
- 기존 RAG, 같은 모델 검증, 재시도, 외부 API 폴백을 품질 모드와 Spec 워크플로에 연결한다.

## 데이터 흐름

1. 사용자 질문 또는 문서를 받는다.
2. RAG 사용 시 BM25로 관련 조각을 검색한다.
3. 로컬 모델이 초안을 생성한다.
4. 같은 모델 검증기와 결정적 규칙이 답변을 검사한다.
5. 실패 시 로컬 모델이 한 번 다시 생성한다.
6. 최종 실패이고 사용자가 폴백을 켠 경우에만 외부 API를 호출한다.
7. 평가와 Spec 산출물을 `/content` 아래 JSON/Markdown으로 저장한다.

## 위험과 대응

- 자기 검증 편향: 결정적 평가 규칙과 외부 평가셋을 병행한다.
- T4 실행 시간: 평가를 체크포인트 방식으로 저장하고 범주·문항 수 필터를 제공한다.
- 프롬프트 인젝션: 업로드 문서의 명령을 데이터로만 취급하고 자동 실행하지 않는다.
- 외부 비용·정보 전송: 폴백 기본 OFF, HTTPS와 Colab 비밀키 사용, 호출 출처 기록.

## 검증 계획

- Python 모듈 단위 테스트
- 75문항 JSON 스키마와 범주 수 검사
- 노트북 JSON 및 모든 Python 코드 셀 구문 검사
- 모델이 없는 로컬 환경에서 fake generator를 사용한 Spec 워크플로 모의 실행
- 실제 T4 모델 전체 실행은 Colab에서 모델 다운로드 후 확인
